СИСТЕМА УПРАВЛЕНИЯ СОТРУДНИКАМИ И УЧЕТА РАБОЧЕГО ВРЕМЕНИ
ПОЛНАЯ ИНФОРМАЦИЯ О ПРОЕКТЕ ДЛЯ ДИПЛОМНОЙ РАБОТЫ

============================================================
1. НАЗНАЧЕНИЕ ПРОЕКТА
============================================================

Проект представляет собой веб-платформу для:
- управления компанией и ее сотрудниками;
- назначения ролей и прав доступа;
- учета рабочего времени (вход/выход, история, текущий статус);
- работы через QR-сканирование (включая режим Scanner Terminal);
- формирования отчетов (Excel и PDF);
- работы в реальном времени (обновления через WebSocket);
- мультиязычного интерфейса (английский/польский) и локализации отчетов.

Основная цель: автоматизировать процессы контроля рабочего времени и управления персоналом в компаниях с ролевой моделью доступа.


============================================================
2. АРХИТЕКТУРА И ОБЩАЯ СТРУКТУРА
============================================================

Тип архитектуры:
- клиент-серверная;
- frontend и backend разделены;
- обмен данными по REST API;
- realtime-обновления через Socket.IO.

Структура репозитория:
- backend/   -> API, бизнес-логика, БД, генерация отчетов
- frontend/  -> пользовательский интерфейс на React
- database/  -> файл базы данных SQLite (company.db)
- assets/    -> статические материалы/скриншоты

Маршрутизация:
- frontend: React Router (SPA маршруты);
- backend: Express Router по модулям (auth, companies, roles, time-logs, reports, users).

============================================================
4. БАЗА ДАННЫХ
============================================================

Тип БД:
- SQLite (через sql.js)
- По умолчанию файл: ./database/company.db
- Путь может задаваться через DB_PATH

Важный момент:
- Да, данные физически хранятся в файле БД внутри проекта (или в пути, который задан переменной окружения).

Основные таблицы:
- users
- companies
- company_members
- roles
- time_logs
- reports
- report_configs
- invitations

Основные связи:
- users <-> companies через company_members
- roles принадлежат company
- time_logs принадлежат user + company
- reports привязаны к company и пользователю-инициатору

Ограничения и целостность:
- foreign keys включены (PRAGMA foreign_keys = ON)
- уникальные поля (например email, qr_code, invite_code)
- уникальность membership (user_id + company_id)

Оптимизации БД (добавлены):
- индекс по time_logs(company_id, clock_in DESC)
- индекс по time_logs(user_id, company_id, clock_out)
- индекс по company_members(company_id, user_id)
- индекс по company_members(user_id, company_id)
- индекс по reports(company_id, created_at DESC)


============================================================
5. БЭКЕНД-МОДУЛИ (КЛЮЧЕВАЯ ЛОГИКА)
============================================================

5.1 Auth
- Регистрация
- Логин
- Получение текущего пользователя (/auth/me)
- JWT-проверка и защита маршрутов
- Rate limit для auth-эндпоинтов

5.2 Companies
- Создание компании
- Вступление по invite code
- Получение данных компании
- Получение списка участников
- Обновление данных компании
- Удаление участника
- Выход из компании
- Удаление компании (owner only)
- Обновление invite code

5.3 Roles/Permissions
- CRUD ролей
- Назначение роли пользователю
- Проверка прав на уровне middleware
- Поддержка implied permissions (родительские права дают дочерние)
- Поддержка legacy-пермишенов и санитизация старых значений

5.4 Time Logs
- Clock in / Clock out
- Получение логов компании
- Получение своих логов
- Текущий статус сотрудника
- Ручное изменение/удаление логов отключено (логи считаются неизменяемыми)

5.5 Reports
- Генерация отчета (excel/pdf)
- Список отчетов
- Скачивание отчетов
- Локализация текста отчета по языку пользователя

============================================================
7. СИСТЕМА ПРАВ ДОСТУПА
============================================================

Характеристика:
- RBAC-подход (Role Based Access Control)
- права хранятся у роли
- владелец компании (owner) имеет полный доступ
- middleware на backend проверяет право перед выполнением действия

Примеры групп прав:
- Отчеты: view_reports, create_reports
- Роли: view_roles, create_roles, edit_roles, delete_roles, manage_roles
- Участники: remove_members, assign_roles, manage_members
- Настройки: view_company_settings, edit_company_settings, manage_company
- Приглашения: view_invite_code, refresh_invite_code
- Учет времени: view_all_timelogs, scan_others, use_scanner_terminal

Implied-права:
- Некоторые "управляющие" права дают дополнительные автоматически.


============================================================
8. REALTIME (SOCKET.IO)
============================================================

Использование:
- Подключение клиента с JWT токеном
- Автоматическое вступление в комнаты компании
- События для обновления интерфейса в реальном времени:
  - employee-clocked-in
  - employee-clocked-out
  - company-invite-code-updated

Назначение:
- Уменьшение ручных обновлений страницы;
- быстрая синхронизация данных между пользователями одной компании.

============================================================
11. ОТЧЕТЫ: ЛОГИКА РАСЧЕТОВ
============================================================

Текущее поведение:
- группировка смен идет по дате clock_in;
- если смена пересекает полночь, она относится к дате входа;
- незакрытые смены (clock_out = null) не считаются завершенными в итоговых часах.

Форматы:
- Excel (xlsx)
- PDF

Поля:
- сотрудник, email, first in, last out, worked time, sessions, break time

============================================================
13. ПРОИЗВОДИТЕЛЬНОСТЬ И ОПТИМИЗАЦИИ
============================================================

Сделанные оптимизации:
- route-based code splitting (ленивая загрузка страниц)
- сокращение fallback polling (socket-first стратегия)
- оптимизация тяжелых вычислений на Dashboard (Map/useMemo/useCallback)
- снижение нагрузки QR-сканирования (интервал сканирования)
- уменьшение лишних runtime-логов в production
- мемоизация фильтрации на Reports
- индексация ключевых SQL-запросов на backend

Эффект:
- быстрее первый рендер;
- меньше лишних API-вызовов;
- плавнее интерфейс при росте данных;
- меньше CPU-нагрузки в сканировании.

============================================================
15. ПРИЧИНЫ ВЫБОРА SQLITE ДЛЯ ДИПЛОМА
============================================================

Преимущества для учебного/дипломного проекта:
- простота установки и запуска;
- не нужен отдельный сервер БД;
- все данные в одном файле;
- удобно демонстрировать и переносить проект.

Ограничения:
- хуже масштабируется под высокую конкурентную запись;
- для крупного production обычно переходят на PostgreSQL.

Вывод:
- Для дипломной работы выбранный вариант обоснован и достаточен.
